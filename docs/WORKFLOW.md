# 医路安（YiLuAn）研发执行流程

## 目标
在不频繁打断用户的前提下，由虚拟团队按工程角色自主推进研发，并对关键决策、阶段性提交、发布结果保留完整轨迹。

---

## 一、执行原则

1. **自主推进**
   - 常规工程事务内部决策、直接执行。
   - 不做无意义确认，不重复向用户请示同类问题。

2. **结果留痕**
   - 每次重要讨论写入 `docs/DECISION_LOG.md`
   - 重大架构决策写入 `docs/adr/`
   - 阶段性工作产出写入对应文档/纪要

3. **阶段提交**
   - 每个阶段必须形成 git commit。
   - 保持提交颗粒度适中，可回滚、可审查。

4. **默认升级条件**
   - 需要用户提供证书、密钥、支付商户号、开发者账号
   - 需要真实发布到生产或应用商店
   - 需要做不可逆高风险操作

---

## 二、标准推进节奏

### Phase 1：方案设计
- 产出：设计文档、接口定义、任务拆解、决策记录
- Commit：`decision:` / `docs:`

### Phase 2：骨架实现
- 产出：新模块、接口骨架、页面骨架、配置项
- Commit：`feat:`

### Phase 3：联调完善
- 产出：完整业务链路、异常处理、边界逻辑
- Commit：`feat:` / `fix:`

### Phase 4：测试收口
- 产出：测试用例、回归结果、问题修复
- Commit：`test:` / `fix:`

### Phase 5：发布准备
- 产出：部署方案、环境变量清单、审核资料、发布记录
- Commit：`ops:` / `docs:`

---

## 三、提交规范

### 推荐格式
`<type>: <summary>`

### 类型
- `feat:` 新功能
- `fix:` 修复问题
- `refactor:` 重构
- `test:` 测试
- `docs:` 文档
- `ops:` 部署/运维
- `decision:` 决策记录

### 示例
- `feat: 接入微信支付下单与回调骨架`
- `fix: 修复订单状态流转与退款边界`
- `test: 增加支付回调幂等测试`
- `ops: 新增 staging 环境部署说明`
- `decision: 冻结 v1 上线范围与发布顺序`

---

## 四、晨会制度

- 每天输出站会纪要到 `Q:\OpenClaw\yiluan-standup\`
- 至少包含：现状、风险、当日优先事项、Action Items
- 若发现关键偏差，可直接调整排期并记录原因
