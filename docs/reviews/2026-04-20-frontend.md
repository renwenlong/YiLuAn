# 微信小程序前端 Review - 2026-04-20

## 1. 总体评价

医路安微信小程序代码库包含 **27 个页面、9 个组件、12 个服务/工具模块**，测试 146 条全部通过。整体架构清晰：统一 API 封装层（含 401 自动刷新队列）、Observer 模式全局 Store、常量集中管理。服务层抽取合理，页面厚度总体可控。但在安全、性能、代码复用、无障碍等方面存在可改进空间，部分页面逻辑过厚需要拆分。

**代码规模统计：**
- 页面 JS 总行数约 3,800+ LOC，组件约 830 LOC，服务层约 670 LOC，工具层约 210 LOC
- 最厚页面：`profile/edit`（350 LOC）、`patient/home`（389 LOC）、`patient/order-detail`（272 LOC）

---

## 2. 亮点

1. **401 并发刷新队列**（`services/api.js`）：多个请求同时 401 时只触发一次 refresh，其余排队等待后重试，解决了常见的竞态问题。
2. **服务层分离到位**：12 个 service 文件覆盖 auth、order、chat、wallet、hospital、review、notification，页面不直接拼 URL。
3. **Observer Store 精简有效**（`store/index.js`，29 LOC）：轻量 pub-sub，支持 unsubscribe 防内存泄漏。
4. **常量集中管理**（`utils/constants.js`）：SERVICE_TYPES / ORDER_STATUS 带 label、color、price 元数据，与后端同步点明确记录在 CLAUDE.md。
5. **安全销户流程**（`settings/delete-account`）：OTP + 勾选确认 + 长按 3 秒计时，多重防误触。
6. **异形屏适配**：两个 tab-bar 组件正确使用 `env(safe-area-inset-bottom)`。
7. **测试覆盖**：146 条 Jest 测试，覆盖 store、services、utils、关键页面逻辑。

## 3. 关键问题

### P0 - 安全 / 功能缺陷

**P0-1: JWT 通过 WebSocket URL 明文传输**
- **位置**：`services/websocket.js:22`、`services/notificationWs.js:19`
- **影响**：Token 出现在 URL query string 中，可能被中间代理日志、浏览器历史记录捕获，存在凭证泄漏风险。
- **方案**：WebSocket 连接建立后通过首条消息发送 token 进行认证，或使用 subprotocol 传递。需后端配合。
- **排期**：1 天（前后端联调）

**P0-2: companion-detail 下单参数名不匹配**
- **位置**：`pages/companion-detail/index.js` 传递 `companionId=`，但 `create-order` 页面期望 `companion_id=`
- **影响**：从陪诊师详情页点击预约后，下单页无法获取陪诊师信息，功能断裂。
- **方案**：统一参数名为 `companion_id`。
- **排期**：0.5 小时

**P0-3: DEV_OTP 硬编码未按环境剥离**
- **位置**：`config/index.js` — `DEV_OTP: '000000'` 在所有环境可见
- **影响**：生产包如果未正确判断环境，可能允许 000000 绕过 OTP 验证。
- **方案**：仅在 `__wxConfig.envVersion === 'develop'` 时导出 DEV_OTP，生产环境返回 undefined。
- **排期**：0.5 小时

### P1 - 性能 / 稳定性

**P1-1: API 请求无超时机制**
- **位置**：`services/api.js` — `wx.request` 未设置 timeout
- **影响**：网络差时请求无限挂起，loading 永远不消失，用户被阻塞。
- **方案**：统一设置 `timeout: 15000`，超时走 reject 路径并 toast 提示。
- **排期**：1 小时

**P1-2: 定时器泄漏（7+ 页面）**
- **位置**：`login`、`patient/order-detail`、`patient/home`、`profile/bind-phone`、`profile/edit`、`settings/delete-account`、`chat/room`
- **影响**：页面快速切换时 setInterval/setTimeout 未清理，导致内存泄漏、幽灵回调更新已销毁页面。
- **方案**：在 onUnload 中集中清理所有 timer，抽取 `useTimer` 工具函数统一管理。
- **排期**：2 小时

**P1-3: 聊天列表 N+1 查询**
- **位置**：`pages/chat/list/index.js` — 对每个会话分别调用 getChatMessages 计算未读数
- **影响**：会话多时产生大量请求，首屏加载慢。
- **方案**：后端提供批量未读数接口（`GET /api/v1/chat/unread-counts`），前端一次获取。
- **排期**：2 天（含后端）

**P1-4: WebSocket / NotificationWs 代码重复**
- **位置**：`services/websocket.js`（94 LOC）与 `services/notificationWs.js`（81 LOC）约 95% 重复
- **影响**：Bug 修复需要改两处，维护成本翻倍。
- **方案**：抽取 `BaseWebSocket` 类，两个模块继承并传入不同 URL 前缀和消息处理回调。
- **排期**：3 小时

### P2 - 体验 / 代码质量

**P2-1: profile/edit 页面过厚（350 LOC）**
- **位置**：`pages/profile/edit/index.js`
- **影响**：患者和陪诊师两套表单逻辑混在一个页面，16 个 state 属性，可读性差。
- **方案**：拆为 `profile/edit-patient` 和 `profile/edit-companion` 两个页面，或提取表单逻辑到 service。
- **排期**：4 小时

**P2-2: 城市选择器逻辑重复**
- **位置**：`pages/patient/home/index.js` 和 `pages/profile/settings/index.js` 各实现一遍（~80 LOC）
- **方案**：抽取为 `components/city-picker` 组件。
- **排期**：2 小时

**P2-3: tab-bar 组件重复**
- **位置**：`patient-tab-bar` 和 `companion-tab-bar` 仅路由映射和标签不同，WXSS 完全相同
- **方案**：合并为单个 `tab-bar` 组件，通过 `role` prop 切换配置。
- **排期**：1.5 小时

## 4. 性能与体验改进项

| 项目 | 现状 | 建议 |
|------|------|------|
| 分包加载 | 未分包，27 页面全在主包 | 按角色分包：patient / companion / profile / chat / legal，主包仅保留 login + role-select |
| 图片 | 无懒加载策略 | 头像/医院图片使用 `lazy-load` 属性，大图用 CDN + webp |
| 长列表 | 分页但无回收 | 订单列表 / 聊天记录超过 100 条时考虑 `recycle-view` 虚拟列表 |
| setData 粒度 | 部分页面整体替换对象 | 对 order-detail 等复杂页面使用路径更新（`this.setData({'order.status': newStatus})`） |
| onShow 过度刷新 | companion/home 每次 onShow 拉 3 个接口 | 加时间戳节流（30s 内不重复拉取），或用 EventChannel 精准刷新 |
| 请求缓存 | 无 | 医院列表、城市列表等静态数据加内存缓存（TTL 5 分钟） |
| wx.reLaunch 开销 | tab-bar 切换用 reLaunch 销毁全部页面 | 权衡：如果不能用原生 tabBar，至少考虑 `wx.redirectTo` + 页面栈管理 |

## 5. 代码质量改进项

| 维度 | 现状 | 建议 |
|------|------|------|
| Lint | 无 ESLint / Prettier | 加 `.eslintrc` + CI 检查，统一 var→const/let |
| 错误处理一致性 | 部分 `.catch(() => {})` 静默吞错 | 统一 `handleError(err, {silent?: boolean})` 工具函数，至少 console.error |
| JSON 解析安全 | websocket onMessage 直接 `JSON.parse` 无 try-catch | 统一封装 `safeParse(data)` |
| 消息 ID | chat/room 用 `Date.now().toString()` | 用 uuid 或 `Date.now() + Math.random().toString(36)` 避免碰撞 |
| word-break | chat-bubble 用 `break-all`（中英混排断字难看） | 改为 `overflow-wrap: break-word` |
| 无障碍 | 全局无 aria-label / role 属性 | rating-stars 至少加 `aria-label="评分 {{value}} 星"` |
| 暗色模式 | 未支持 | 短期：variables.wxss 增加 `@media (prefers-color-scheme: dark)` 基础适配 |
| rating-stars 半星 | 仅支持整数 0-5 | 如果后端返回小数评分（如 4.5），展示端应支持半星渲染 |
| token.js Base64 | 手写 Base64 解码器（O(n²)） | 微信环境可用 `ArrayBuffer` + `TextDecoder`，或直接 `atob()` |
| 同步存储 | `wx.setStorageSync` / `getStorageSync` 阻塞主线程 | 非关键路径改用异步版本 `wx.setStorage` |
