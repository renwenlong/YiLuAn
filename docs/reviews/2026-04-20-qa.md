# QA Review - 2026-04-20

## 1. 总体评价

医路安项目当前测试基础已具备一定规模，但从 QA 专业视角看仍有显著改进空间。

**覆盖率快照（2026-04-20）：**

| 平台 | 测试数 | 覆盖率 | 框架 |
|------|--------|--------|------|
| 后端 (Python) | 539 passed / 5 deselected / 1 xfailed | **80.0%** 行覆盖 | pytest + pytest-asyncio |
| 小程序 (WeChat) | 146 passed | 未度量 | Jest |
| iOS | 0 | 0% | 未配置 |

**优势：** 后端覆盖率从 71.9% 提升至 80%，测试用例从 470 增至 539；小程序从 77 增至 146 个测试；CI 已包含后端 + 小程序 + Docker 构建验证。

**主要风险：** 无分支覆盖率、无性能/安全测试、iOS 零测试、小程序无覆盖率度量、E2E 场景不完整、无契约测试保障前后端一致性。

## 2. 测试金字塔现状分析

### 后端（539 tests）

| 层级 | 测试文件 | 用例数（估） | 比例 |
|------|---------|-------------|------|
| 单元测试（service/util 直接调用） | `services/test_order.py`, `test_sms_providers.py`, `test_pii.py`, `utils/test_outbound.py` 等 | ~120 | 22% |
| 集成测试（HTTP client → full stack） | `test_orders.py`, `test_auth.py`, `test_chats.py`, `test_reviews.py` 等 | ~380 | 71% |
| E2E / 生命周期 | `test_full_lifecycle.py`, `test_refund_e2e.py` | ~32 | 6% |
| 冒烟测试（真 PG + Alembic） | `smoke/test_pg_alembic_smoke.py`, `smoke/test_readiness_blocker.py` | ~5 | 1% |

**问题：金字塔倒置趋势。** 集成测试占 71%，单元测试仅 22%。集成测试通过 httpx ASGITransport 走完整 HTTP → Router → Service → Repo → SQLite 链路，运行较慢（539 tests / 69s ≈ 0.13s/test）。理想比例应为单元 60% / 集成 30% / E2E 10%。

### 小程序（146 tests）

主要是 service 层 API 调用的 mock 测试（`wx.request` mock），覆盖了 HTTP 请求参数构造和返回值处理。有少量页面级测试（6 页面）和组件测试（1 个）。无真机测试、无 UI 自动化测试。

## 3. 关键问题

### P0 — 阻断性 / 高风险

| # | 问题 | 影响 | 建议 |
|---|------|------|------|
| P0-1 | **无分支覆盖率** | 80% 行覆盖率掩盖了大量未测试分支（如 if/else、异常路径）。订单状态机 `ORDER_TRANSITIONS` 的非法转换路径可能未充分覆盖 | 启用 `--cov-branch`，目标分支覆盖 ≥65% |
| P0-2 | **iOS 零测试** | 三端发布，iOS 端无任何质量保障。SwiftUI ViewModel 逻辑、网络层、状态管理均无验证 | 至少补充 ViewModel 单元测试和 APIClient mock 测试 |
| P0-3 | **支付流程缺真实验证** | Mock payment 始终成功，从未测试过支付失败、超时、金额不匹配等异常场景的端到端行为 | 补充 mock 支付失败场景；上线前必须有沙箱环境集成测试 |

### P1 — 重要

| # | 问题 | 影响 |
|---|------|------|
| P1-1 | **小程序覆盖率未度量** | 146 个测试但不知道覆盖了多少代码，可能存在大量测试盲区 |
| P1-2 | **无契约测试** | 前后端 API 接口、WebSocket 消息格式无自动化一致性验证，CLAUDE.md 中明确标注了 4 个同步点但无自动保障 |

## 5. 缺失的测试类型

| 测试类型 | 当前状态 | 必要性 | 建议 |
|---------|----------|--------|------|
| **契约测试** | 无 | 高 | 用 `schemathesis` 或自定义 schema 校验保障 API 契约；用 JSON Schema 校验 WS 消息格式 |
| **性能测试** | 无 | 中高 | 关键接口（订单列表、医院搜索、WS 并发）需 locust/k6 基准测试，设定 P95 延迟阈值 |
| **安全测试** | 无 | 高 | JWT 伪造/过期、越权访问（患者访问他人订单）、SQL 注入、PII 泄露检测 |
| **回归测试自动化** | 部分（CI 跑全量） | 中 | 需标记冒烟子集用于快速回归，完整回归放 nightly |
| **兼容性测试** | 无 | 中 | 小程序需覆盖微信版本 ≥7.0、iOS/Android 差异；后端需 PG 14/15 兼容 |
| **负载测试** | 无 | 中 | 陪诊师抢单并发、支付回调高频场景需验证数据一致性 |
| **混沌测试** | 无 | 低 | 数据库断连、Redis 不可用时的降级行为 |
| **无障碍测试** | 无 | 低 | 小程序 ARIA 标签、iOS VoiceOver 支持 |

## 6. 测试基础设施改进建议

### 6.1 测试数据管理
- **引入 factory 模式**：使用 `factory_boy` 或自建 builder 替代硬编码 fixture。当前 `conftest.py` 中 `seed_user`、`seed_order` 等 fixture 包含大量硬编码值，新增测试场景需要复制修改。
- **测试隔离**：当前每个测试重建全部表（`autouse` fixture），开销较大。考虑 savepoint 策略减少表重建次数。

### 6.2 CI/CD 增强
- 当前 CI (`test.yml`) 只跑 `pytest -v`，未生成覆盖率报告。建议：
  - 加 `--cov --cov-report=xml` + Codecov/Coveralls 集成
  - 设置覆盖率门禁（PR 不得降低覆盖率）
  - 拆分快速测试（< 10s）和完整测试（冒烟 + PG 集成）

### 6.3 FakeRedis 升级
- 自研 `FakeRedis` 缺少 TTL 自动过期、pipeline、pub/sub ACK。建议切换到 `fakeredis[aioredis]` 库（PyPI），保真度更高且维护成本为零。

### 6.4 小程序测试提升
- 配置 Jest `--coverage` 输出覆盖率
- 增加页面级 snapshot 测试（当前仅 6 页面有测试 / 共 27 页面）
- 考虑引入 `miniprogram-automator` 进行真机 UI 自动化

### 6.5 测试文档
- 缺少 test plan 文档、测试命名规范、bug → 测试用例转化流程
- 无手工测试 checklist（UAT 场景未文档化）
