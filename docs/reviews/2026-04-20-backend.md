## 1. 总体评价

后端采用 FastAPI + SQLAlchemy 2.0 + PostgreSQL 的异步架构，分层清晰（Router → Service → Repository → Model），依赖注入通过 `Annotated` 类型实现，整体代码质量在 MVP 阶段属上乘。核心订单状态机 (`ORDER_TRANSITIONS`) 设计合理，支付幂等方案（SAVEPOINT + unique constraint）工程水准高，测试体系 42 个文件 / 11000+ 行覆盖主链路。

主要风险集中在：**OTP 验证无尝试次数限制**（P0 安全）、**退款回调未闭环**（P1 功能缺失）、**check-expired 端点无鉴权**（P1 安全）、以及若干性能隐患（N+1 查询、Float 金额精度）。建议在进入生产前优先修复 P0/P1 项。

## 2. 代码亮点

1. **支付幂等设计精良** — `PaymentCallbackLog` 表 + `(provider, transaction_id)` unique constraint + SAVEPOINT 嵌套事务，防止重复回调修改业务状态。`record_callback_or_skip()` 是教科书级实现。

2. **订单状态机明确可审计** — `ORDER_TRANSITIONS` 字典定义合法转换，`_validate_transition()` 统一校验，`OrderStatusHistory` 记录每次变迁，timeline 可回溯。

3. **Provider 抽象可插拔** — 支付 (mock/wechat) 和短信 (mock/aliyun/tencent) 均通过 Factory + 基类抽象，`settings.payment_provider` 一键切换，测试时零外部依赖。

4. **测试基础设施成熟** — 工厂模式 seed fixtures、autouse 的 `setup_database` 每测试隔离、FakeRedis、ASGITransport 全栈集成测试，无明显 flaky 测试。

5. **配置验证严格** — `config.py` 在 production 环境强制校验 JWT 密钥非默认值、DEBUG=false、WeChat/SMS 凭证齐全，防止带着开发配置上线。

## 3. 关键问题

### P0 — OTP 验证码暴力破解

- **问题**: `auth.py:65` `verify_otp()` 无尝试次数限制。6 位纯数字验证码仅 100 万种组合，攻击者可高频枚举。
- **影响**: 任意手机号可被登录/绑定，用户账号被劫持。
- **方案**: 在 Redis 中记录 `otp:attempts:{phone}` 计数，每次 verify 递增，5 次失败后锁定并删除 OTP。
- **排期**: 1 天，阻塞上线。

### P1 — 退款回调未更新 Payment 状态

- **问题**: `payment_callback.py:189` 标注 `TODO: update refund Payment record status based on refund_status`，退款回调仅记录日志，不更新 `Payment.status`。
- **影响**: 退款 Payment 永远停留在 `pending`，前端/对账无法区分退款成功与失败。
- **方案**: 在 `PaymentService` 中添加 `handle_refund_callback()` 方法，根据 `refund_status` 更新退款记录状态。
- **排期**: 2 天。

### P1 — check-expired 端点无鉴权

- **问题**: `api/v1/orders.py:154` `POST /orders/check-expired` 无任何 auth 依赖，任何人可调用。
- **影响**: 恶意调用可批量过期订单，或作为探测接口。
- **方案**: 添加 admin token 校验或内部调度专用 header。
- **排期**: 0.5 天。

### P1 — User 模型 role/roles 双字段冗余

- **问题**: `models/user.py` 同时存在 `role: Enum` 和 `roles: str` 两个字段。`role` 是旧字段，`roles` 是新的逗号分隔字符串。依赖层混用两者 (`dependencies.py` 读 `role`，JWT payload 同时带 `role` 和 `roles`)。
- **影响**: 角色判断分散在多处逻辑，新增角色时容易遗漏同步。
- **方案**: 迁移统一到 `roles` 字段，废弃 `role` 列（需 migration + 全局替换）。
- **排期**: 3 天。

### P2 — 自动退款 silent failure

- **问题**: `order.py:290-298`、`order.py:410-418`、`order.py:440-448` 三处自动退款逻辑均 `except BadRequestException: pass`，退款失败完全静默。
- **影响**: 用户取消/拒单后以为已退款，实际资金未到账，无告警。
- **方案**: 记录退款失败日志 + 写入 notification 或告警队列，让运营可介入。
- **排期**: 1 天。

### P2 — auth.py 中 OTP 验证逻辑重复

- **问题**: `verify_otp()` (line 65-77) 和 `bind_phone()` (line 163-175) 包含几乎相同的 OTP 校验代码块。
- **影响**: 修改校验逻辑需同步两处，容易遗漏。
- **方案**: 提取 `_verify_otp_code(phone, code)` 私有方法。
- **排期**: 0.5 天。

## 4. 性能隐患

### 4.1 list_orders N+1 查询

`order.py:166-167` 对每条订单逐一调用 `_fill_payment_status()`，每次执行一条 `SELECT ... FROM payments WHERE order_id = ?`。当分页 20 条时产生 20 次额外查询。

**方案**: 批量查询 `SELECT order_id, payment_type FROM payments WHERE order_id IN (:ids)`，一次填充。

### 4.2 金额使用 Float 类型

`Order.price` 和 `Payment.amount` 均为 `Float`。浮点运算在退款 50% 时 (`round(order.price * 0.5, 2)`) 可能出现精度问题（如 299.0 * 0.5 = 149.5 无问题，但其他金额组合可能出现 .0000001 误差）。

**方案**: MVP 阶段可接受，生产环境建议迁移到 `Numeric(10, 2)` 并使用 `Decimal`。

### 4.3 WebSocket 每消息创建 Session

`ws.py:202` 每条聊天消息 `async with async_session() as session:` 创建新连接。高频聊天场景下连接池压力大。

**方案**: 可在 WebSocket 连接建立时创建一个长生命周期 session，或批量写入。

### 4.4 companion 广播查询无缓存

`order.py:337-340` 支付后广播通知时查询 `companion_repo.search(hospital_id=..., limit=50)`，每次支付都查库。

**方案**: 对活跃 companion 列表做短 TTL Redis 缓存。

## 5. 安全隐患

### 5.1 OTP 暴力破解 (P0，详见第 3 节)

### 5.2 JWT token 在 WebSocket URL 中传递

`ws.py` 通过 `?token=<jwt>` 传递认证信息。JWT 会出现在服务器访问日志、代理日志、浏览器历史中。

**缓解**: WebSocket 场景下难以避免（HTTP header 在握手前不可设置），但应确保日志脱敏，且 token 有效期短。

### 5.3 requirements.txt 版本未完全锁定

- `cryptography>=43.0.0` — 仅下限，可能引入破坏性更新
- `alibabacloud-dysmsapi20170525>=3.0.0` / `tencentcloud-sdk-python-sms>=3.0.0` — 同上
- `APScheduler>=3.10,<4.0` — 宽范围

**方案**: 使用 `pip-compile` 生成锁文件，或至少固定到 `==` 版本。

### 5.4 WeChat 支付证书手动管理

`providers/payment/wechat.py` 中证书缓存使用 `threading.Lock`（在 async 上下文中可能短暂阻塞事件循环），且证书轮换标注 TODO 未实现。

**方案**: 替换为 `asyncio.Lock`；实现自动从 `/v3/certificates` 拉取平台证书。

### 5.5 Admin 认证为硬编码 token

`admin_audit.py` 使用环境变量中的固定字符串做 admin 鉴权，无过期/轮换机制。

**缓解**: MVP 可接受，生产前需迁移到 OAuth2 或独立 admin JWT。
