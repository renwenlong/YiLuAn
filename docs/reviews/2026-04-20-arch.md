# 架构 Review - 2026-04-20

## 1. 总体评价（7/10）

当前架构在 MVP 阶段完成度较高：清晰的四层分层（API → Service → Repository → Model）、可插拔支付 provider、显式订单状态机、完善的文档体系（28 条 Decision Log + 11 条技术债跟踪）。80% 测试覆盖率为后续重构提供了安全网。主要风险在于 OrderService 已成为 God Service（565 行、9 个依赖），以及支付-订单状态联动存在已知缺口（TD-PAY-01），需要在 WeChat Pay 上线前修复。

## 2. 设计亮点

### 2.1 显式订单状态机 + 变更历史审计

`ORDER_TRANSITIONS` 字典将状态转换规则集中定义在 `models/order.py`，而非散落在各个 service 方法中。`OrderStatusHistory` 模型记录每一次状态变更的发起人和时间戳，为运营排查、合规审计提供了完整链路。`_validate_transition()` 作为唯一校验入口，保证了状态机的一致性。这个设计在 MVP 阶段已经足够好。

### 2.2 可插拔支付 Provider + 编排层分离

`PaymentProvider` 基类定义了 `create_order/verify_callback/refund/query` 四个接口，`MockPaymentProvider` 和 `WechatPaymentProvider` 各自实现。`PaymentService` 作为编排层持有 Provider 实例，负责幂等性（`PaymentCallbackLog` + SAVEPOINT）、退款记录持久化等横切逻辑。Provider 不依赖 ORM 模型（通过 `OrderDTO/RefundDTO` 解耦），扩展新支付渠道（如支付宝）只需新增 provider + factory 分支，不影响上层。

### 2.3 WebSocket Pub/Sub 多副本架构

`WsPubSubBroker` 支持 Redis Pub/Sub 跨副本消息广播，同时内置降级到进程内模式的 fallback。通知和聊天分为两个独立 broker（`user_id` 维度 vs `order_id` 维度），关注点分离清晰。连接数上限（`ws_max_connections_per_user`）防止单用户资源耗尽。这套设计在不引入消息队列的前提下，已经能支撑中等规模的实时需求。

### 2.4 文档驱动的决策体系

28 条 Decision Log + ADR 模板 + 技术债追踪 + UAT checklist，形成了从决策到执行到验收的闭环。特别是 `TECH_DEBT.md` 按模块分类、标注优先级、记录缓解方案，使得技术债可见可管理，而非散落在代码注释中。

## 3. 关键问题

### P0-1 OrderService 已成为 God Service

- **问题描述**：`OrderService.__init__` 注入 8 个依赖（order_repo、hospital_repo、payment_repo、history_repo、companion_repo、notification_svc、payment_svc、chat_repo），565 行代码承载了订单创建、支付发起、退款、过期检查、状态流转、通知触发、聊天消息创建等职责。每次新增功能（如部分退款策略调整、新的通知类型）都需要修改这个类。
- **影响范围**：所有订单相关功能的开发速度和测试复杂度。当前 `order.py` 测试覆盖率 88% 已经需要 69+ 用例维持，再加 1-2 个功能（如优惠券、多次部分退款）会失控。
- **建议方案**：
  - **方案 A — 领域事件解耦**：引入轻量级事件总线（如 in-process `EventBus`），订单状态变更时发布事件（`OrderAccepted`、`OrderCancelled`），NotificationService 和 ChatService 订阅相应事件自行处理。Trade-off：增加间接性，调试需要追踪事件链路。
  - **方案 B — 拆分为 OrderCommandService + OrderQueryService + RefundService**：将退款逻辑（含自动退款策略）拆到 `RefundService`，查询逻辑拆到只读 service。Trade-off：简单直接但仍是过程式，不如事件驱动灵活。
- **建议排期**：本 sprint（在 WeChat Pay 上线前完成，否则退款逻辑的修改会更困难）

### P0-2 支付-订单状态联动缺口（TD-PAY-01）

- **问题描述**：`check_expired_orders` 将订单标为 expired 时，不会将 pending 的 Payment 行降级为 failed。若 WeChat Pay 回调在订单过期后到达，`handle_pay_callback` 会将 pay 翻为 success，但不会触发退款——用户付了钱，订单却是过期状态，资金悬空。
- **影响范围**：上线 WeChat Pay 后的所有过期订单场景。Mock 环境下因即时成功不触发，但生产环境中网络延迟可能使回调在数小时后才到达。
- **建议方案**：
  - **方案 A — 过期时主动关单**：`check_expired_orders` 在标 expired 后调用 `provider.close_order(trade_no)` 关闭微信侧待支付单，同时将 Payment 行标为 failed。Trade-off：需要 provider 实现 `close_order` 接口。
  - **方案 B — 回调时防御性退款**：`handle_pay_callback` 在处理成功回调前检查订单状态，若订单已 expired/cancelled，立即触发全额退款。Trade-off：用户可能看到"支付成功"后又收到退款，体验稍差。
- **建议排期**：本周（阻塞 WeChat Pay 上线）

### P1-1 WebSocket ws.py 直接操作 Repository，绕过 Service 层

- **问题描述**：`websocket_chat` 端点直接 `async with async_session()` 创建 session，实例化 `OrderRepository` 和 `ChatMessageRepository` 进行读写，跳过了 Service 层。这破坏了分层架构约定（API → Service → Repository），使得聊天消息的业务规则（如禁言、内容审核）无法在 Service 层统一拦截。
- **影响范围**：聊天功能的可维护性。未来加入内容审核、敏感词过滤、消息频率限制时，需要在 WebSocket handler 里直接写业务逻辑，与 REST API 的聊天接口形成两套代码。
- **建议方案**：
  - **方案 A — 聊天消息走 ChatService**：WS handler 只负责连接管理和消息收发，消息持久化和广播委托给 `ChatService.send_message(order_id, user_id, content)`。Trade-off：每条消息多一层调用开销，但可忽略不计。
  - **方案 B — 保持现状，加注释标记**：当前 MVP 阶段聊天逻辑简单（纯文本存储+广播），分层收益不大。在代码注释中标记 TODO，等功能复杂时再重构。Trade-off：技术债累积。
- **建议排期**：本 sprint

### P1-2 float 类型存储金额

- **问题描述**：`Order.price` 和 `Payment.amount` 使用 `Float` 类型，所有金额计算（如 50% 退款 `round(order.price * 0.5, 2)`）使用浮点运算。IEEE 754 浮点数无法精确表示十进制小数，可能导致金额不一致（如 ¥149.00 * 0.5 = 74.5 没问题，但更复杂的折扣计算可能出现 0.1 + 0.2 ≠ 0.3 类问题）。
- **影响范围**：支付和退款的所有金额字段。当前固定价格 (149/199/299) 和简单的 50%/100% 退款策略下问题不大，但引入优惠券、阶梯折扣后风险急剧上升。
- **建议方案**：
  - **方案 A — 改用 Decimal/Numeric**：DB 字段改为 `Numeric(10, 2)`，Python 层用 `decimal.Decimal`，所有运算精确。Trade-off：需要 migration + Pydantic schema 调整 + 前端适配。
  - **方案 B — 分转存储**：所有金额以分（int）存储，展示层除以 100。Trade-off：改动范围大，但微信支付本身就以分为单位，可以消除转换误差。
- **建议排期**：下 sprint（与优惠券功能一起做）

### P1-3 用户角色 comma-string 模型

- **问题描述**：`User.roles = "patient,companion"` 以逗号分隔字符串存储角色。当前仅两种角色问题不大，但无法支持 RBAC 需求（如 admin 子角色、城市运营）。`user.role == UserRole.patient` 的比较逻辑在多角色用户上语义不清——代码中有时用 `user.role`（取第一个角色），有时用 `has_role()`，不一致。
- **影响范围**：鉴权逻辑的正确性。`OrderService.cancel_order` 中 `user.role == UserRole.patient` 对于同时拥有 patient 和 companion 角色的用户可能判断错误。
- **建议方案**：
  - **方案 A — 角色枚举数组**：PostgreSQL `ARRAY(String)` 或枚举数组，配合 `@hybrid_property` 保持 API 兼容。Trade-off：migration 成本中等。
  - **方案 B — 请求级角色上下文**：登录时选择当前角色写入 JWT claims，service 层从 token 读取而非从 User model。Trade-off：需要前端配合切换角色时重新获取 token。
- **建议排期**：下 sprint

### P2-1 缺少统一的结构化日志和请求追踪

- **问题描述**：当前日志使用标准 `logging.getLogger(__name__)`，无统一的 JSON 格式、无 request_id / trace_id 传递。`docs/observability.md` 已定义了结构化日志规范但尚未落地。生产环境排查问题时无法将一次请求的 API → Service → Repository → Provider 日志串联。
- **影响范围**：生产可观测性。当前开发阶段影响不大，但上线后每次排查问题都需要手动关联日志。
- **建议方案**：
  - **方案 A — FastAPI middleware 注入 request_id**：middleware 从 header 读取或生成 `X-Request-ID`，写入 `contextvars`，logging filter 自动附加到每条日志。Trade-off：简单易实现，10-20 行代码。
  - **方案 B — OpenTelemetry 全套**：按 `observability.md` 的 P2 计划引入 OTLP exporter。Trade-off：依赖重，MVP 阶段可能过度。
- **建议排期**：本 sprint（方案 A 即可）

## 5. 演进路线建议

### 近期（1-2 个月）：稳固核心，上线微信支付

首要任务是修复 TD-PAY-01 并完成 OrderService 拆分，确保支付链路的正确性和可维护性。将退款逻辑独立为 RefundService，引入轻量级领域事件解耦通知和聊天。金额字段迁移到 Decimal，为优惠券等促销功能打基础。落地 request_id 日志中间件，为生产排查提供最基本的追踪能力。目标：WeChat Pay + 真实 SMS 上线，覆盖率保持 80%+。

### 中期（3-4 个月）：扩展业务能力，提升运营效率

引入运营后台配置能力：服务类型和价格可配置化（从 DB 读取而非硬编码）、优惠券/促销模块、陪诊师结算与提现流程。用户角色模型升级支持 RBAC，为城市运营人员、客服等角色提供细粒度权限。WebSocket 层完善内容审核和消息回补。考虑引入异步任务队列（Celery/arq）替代 APScheduler，为更复杂的定时任务（如结算、对账）提供可靠保证。

### 远期（5-6 个月）：业务多元化，架构弹性化

若要扩展视频问诊、药品配送等新业务线，当前以 Order 为中心的模型需要演进为更通用的「服务订单」抽象。建议采用 DDD 的限界上下文思路：将 Order、Payment、Chat、User 拆分为独立的领域模块，通过事件驱动通信。API 网关层考虑引入 rate limiting、API versioning。数据层面，随着订单量增长，历史订单和聊天记录需要归档策略（冷热分离）。监控体系按 `observability.md` 规划落地 Prometheus metrics + Grafana dashboards。
