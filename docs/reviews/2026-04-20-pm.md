# 产品 Review & 竞品调研 - 2026-04-20

## Part 1: 现有产品 Review

### 1.1 总体评价

医路安 WeChat 小程序已完成 MVP 核心闭环：患者注册 → 选医院 → 选陪诊师 → 下单 → 支付 → 服务中沟通 → 评价。陪诊师端也已覆盖注册 → 接单 → 服务 → 收入管理。整体架构清晰，27 个页面覆盖了双角色（患者/陪诊师）主流程。

**优势：**
- 双角色一套小程序，通过角色切换减少用户管理成本
- 城市+医院+服务类型三级筛选，定位精准
- 订单状态机设计合理，支持取消/退款分支
- WebSocket 实时聊天已跑通基础链路

**核心短板：**
- 钱包充值/提现功能为 stub（"功能开发中"），支付闭环未完成
- 多处导航链路断裂（详见 1.4）
- IM 仅支持文本，缺少图片/语音/位置等医疗场景高频消息类型
- 评价体系单一（仅 1-5 星 + 文字），缺少维度评分
- 通知系统无深链跳转，用户点击通知后无法直达相关订单

### 1.2 用户旅程分析

按主流程：**发现 → 决策 → 下单 → 服务 → 评价 → 复购** 分析：

#### 发现阶段
- **入口**：患者首页提供 GPS 定位 + 城市选择 + 关键词搜索，医院筛选体验良好
- **问题**：无推荐算法，无热门医院/热门陪诊师运营位；新用户进入首页后缺少引导，不了解"陪诊"是什么服务
- **缺失**：无新手引导页/服务说明页，无首页 Banner/运营活动入口

#### 决策阶段
- **陪诊师列表**：展示评分、接单量、简介，曾服务过的陪诊师优先排序（好设计）
- **陪诊师详情页**：有评价列表、服务区域、经验信息
- **问题**：
  - 详情页评价空状态判断有 bug（`wx:if` 未正确守卫 `reviews.length === 0`），导致空状态提示与评价列表同时显示
  - 从陪诊师详情页点击"预约陪诊"时传参为 `companionId`，但下单页读取 `companion_id`，**导致陪诊师预选失败**（高优 bug）
  - 无陪诊师对比功能，无收藏/关注功能
  - 缺少陪诊师资质证书展示（仅文字描述）

#### 下单阶段
- **流程**：服务类型+价格（固定）→ 医院（固定）→ 选陪诊师（可换）→ 日期 → 时间 → 备注 → 确认下单
- **问题**：
  - 未绑定手机号时会跳转绑定页，但绑定完成后的回跳体验需验证
  - 时间选择为自由时间选择器，无医院营业时间约束
  - 无"为家人下单"功能（仅能为自己下单）
  - 三种服务类型价格固定（¥149/199/299），无动态定价能力

#### 服务阶段
- **订单状态流转**：created → accepted → in_progress → completed，支持多节点取消
- **退款策略**：未支付取消全退，进行中取消退 50%（合理）
- **沟通**：WebSocket 实时聊天，订单维度绑定
- **问题**：
  - 聊天仅支持文本，医疗场景下图片（检查报告）、位置（医院内导航）是刚需
  - 陪诊师端"通知患者开始服务"是 request-start 动作，需患者确认——但患者可能在候诊中不方便操作，流程可优化
  - "进行中"订单列表未包含 `accepted` 状态订单，导致已接单但未开始的订单在"进行中"tab 不可见
  - 拨打电话按钮依赖 `order.companion.phone` / `order.patient.phone`，但 API 响应未填充嵌套对象，**电话按钮无效**

#### 评价阶段
- 仅患者可评价陪诊师，单向评价
- 1-5 星 + 文字（≥5 字），简单但足够 MVP
- **缺失**：无图片评价、无维度评分（守时/专业/沟通）、无匿名选项、评价后不可编辑

#### 复购阶段
- 订单详情有"重新下单"功能（rejected/expired 状态），复用原订单参数
- **缺失**：无复诊提醒、无历史陪诊师快捷重约、无优惠券/会员体系拉动复购

### 1.3 现有功能优缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 注册登录 | 微信一键 + 手机号 OTP 双通道 | 可跳过手机绑定，后续下单时才拦截，体验断裂 |
| 医院筛选 | GPS 定位 + 城市选择 + 搜索，体验流畅 | 无医院详情页，无科室信息 |

| 支付 | Mock 支付已跑通 | 钱包充值/提现为 stub |
| 聊天 | WebSocket 实时、已读状态 | 仅文本、无图片/语音/位置 |
| 评价 | 简洁易用 | 单维度、无图片、不可编辑 |
| 通知 | 有列表、已读标记、分页 | 无深链跳转、无推送（小程序订阅消息未接入） |
| 钱包 | UI 已完成 | 充值和提现均为 stub |

### 1.4 P0 必修体验问题（≥3 条）

**P0-1: 陪诊师详情页预约链接参数错误（高优 bug）**
- 位置：`pages/companion-detail/index.js` line 57
- 问题：传参 `companionId` 但下单页读 `companion_id`，导致从详情页预约时陪诊师未被预选
- 影响：核心转化路径断裂，患者需手动重选陪诊师

**P0-2: 陪诊师端通知页导航路径错误**
- 位置：`pages/companion/home/index.js` line 97
- 问题：跳转 `/pages/profile/notifications/index`（不存在），正确路径为 `/pages/notification/index`
- 影响：陪诊师无法查看通知，可能错过接单提醒

**P0-3: 电话拨打功能无效**
- 位置：`patient/order-detail/index.js` 和 `companion/order-detail/index.js`
- 问题：`onCallCompanion`/`onCallPatient` 依赖 `order.companion.phone`/`order.patient.phone`，但 API 响应未填充嵌套对象
- 影响：服务过程中患者/陪诊师无法电话联系，紧急情况下的沟通通道断裂

**P0-4: 钱包充值/提现未实现**
- 位置：`pages/profile/wallet/index.js` lines 79, 87
- 问题：按钮点击仅 toast "功能开发中"
- 影响：支付闭环不完整，陪诊师无法提现收入

**P0-5: "进行中"订单 tab 遗漏 accepted 状态**
- 位置：`pages/orders/index.js`
- 问题：`accepted` 状态订单既不在"待接单"也不在"进行中"，用户找不到
- 影响：已接单订单"消失"，造成用户困惑

### 2.3 各竞品深度卡片

#### 春雨医生
春雨医生是国内最早的互联网医疗平台之一，核心能力在在线问诊（图文、电话、视频），拥有 7×24 小时服务和"60 秒极速回复"的招牌体验。平台积累了大量医生资源和用户健康数据。目前未观察到独立的"陪诊"服务模块，但其"找医院""就医攻略"等内容功能与陪诊前链路有协同空间。对医路安的启示：**春雨的内容运营（疾病科普、就医攻略）值得借鉴**，可在陪诊前通过内容建立用户信任。[来源: chunyuyisheng.com · 2026-04]

#### 京东健康
京东健康依托京东生态，在药品电商、在线问诊、体检等领域布局完整。陪诊服务作为"健康服务"板块的子功能存在，但并非核心业务。其优势在于：①京东物流支撑的药品配送能力 ②企业客户健康管理 ③支付和会员体系完善。对医路安的启示：**药品代购代送是天然延伸服务**，京东健康的药品配送模式值得参考。待补充调研：具体陪诊功能细节、价格、服务流程。

#### 平安好医生（平安健康）
平安好医生背靠平安保险集团，定位"管理式医疗+家庭医生"。陪诊服务主要面向保险客户的增值服务——高净值客户可享免费陪诊、名医预约等。这种模式下陪诊是获客/留存工具而非独立盈利业务。B 站从业者信息显示，部分陪诊师的单源来自保险公司派单。对医路安的启示：**B 端合作（保险公司、企业健康管理）是重要的获客渠道**，可考虑 API 对接模式。[来源: bilibili.com · 2025]

#### 微医
微医是国内拥有最多互联网医院牌照的平台，核心能力在在线挂号、远程问诊、医保在线结算。陪诊相关服务（代问诊、代取报告等）在部分城市可用。其优势在于与医院 HIS 系统的深度对接。对医路安的启示：**与医院系统对接（挂号、排队提醒、报告推送）是长期差异化方向**。待补充调研：具体陪诊功能和价格。

#### 个体陪诊师群体
根据 B 站 2025 年从业者分享视频，个体陪诊师是目前市场的重要供给方。主要特征：①通过自媒体（抖音、小红书）获客 ②微信/电话直接对接 ③价格 ¥200-800/次不等（一线城市偏高）④收入不稳定（"饿不死但发不了财"）⑤上海已推出专项职业能力证书（约 ¥980 考证费，人社部门主导）⑥保险公司开始派单合作。痛点：获客难、收入波动大、缺乏平台保障、无标准化服务流程。对医路安的启示：**平台化是个体陪诊师的刚需，解决获客和信任问题是核心价值主张**。[来源: bilibili.com · 2025]

#### 其他观察
- **滴滴陪诊**：未找到相关信息，大概率不存在独立业务
- 陪诊行业仍处于早期，垂直平台少，多数为综合医疗平台的子功能
- 上海人社局已培训 1200+ 陪诊师并推出专项证书，政策信号积极 [来源: bilibili.com · 2025]

### 3.2 高优先级功能详细方案

#### P0-A: IM 富媒体消息（图片/语音/位置）

**背景**：当前聊天仅支持文本。医疗陪诊场景下，患者需要发送检查报告图片、医保卡照片等；陪诊师需要发送医院内位置帮助患者定位。语音消息对老年用户更友好。

**功能范围**：
- 图片消息：从相册选择或拍照，支持预览和全屏查看，图片压缩上传（≤2MB）
- 语音消息：长按录音（≤60s），气泡播放，自动转文字（可选）
- 位置消息：发送当前位置 / 选择地点，接收方可打开地图导航

**技术方案**：
- 前端：扩展 chat/room 页面，新增消息类型渲染（image/voice/location），接入 `wx.chooseMedia`、`wx.startRecord`、`wx.getLocation`
- 后端：新增文件上传 API（OSS/MinIO），WebSocket 消息协议扩展 type 字段
- 存储：聊天图片/语音存 OSS，消息记录存 DB

**验收标准**：患者可发送检查报告图片，陪诊师可发送位置，双方可语音沟通

**工时**：前端 1 周 + 后端 1 周 = 2 周

#### P0-B: 代他人下单（家属端）

**背景**：陪诊最大用户场景是"子女为年迈父母预约陪诊"。当前系统仅支持为自己下单，这是严重的功能缺失。

**功能范围**：
- 下单页增加"就诊人"选择：自己 / 已保存的家人
- 家人管理模块：添加家人（姓名、手机号、关系、医保类型、常见病史）
- 订单详情展示就诊人信息（非下单人）
- 陪诊师端可看到实际就诊人信息

**技术方案**：
- 新增 `Patient` 模型（与 User 一对多），字段：name, phone, relationship, medical_notes
- 下单页增加就诊人选择 UI，`create-order` API 扩展 `patient_profile_id` 字段
- 个人中心新增"我的家人"管理页面
- 订单详情页展示就诊人信息

**验收标准**：用户可添加家人并为家人下单，陪诊师能看到实际就诊人信息

**工时**：前端 1 周 + 后端 0.5 周 = 1.5 周

#### P1-A: 复诊提醒 / 随访

**背景**：慢病患者需定期复诊，陪诊后设置提醒可带动复购。当前无任何主动触达用户的机制。

**功能范围**：
- 订单完成后弹窗："是否设置复诊提醒？"
- 提醒管理页：查看/编辑/删除已设提醒

**技术方案**：

## Part 4: 战略建议

### 差异化定位

当前陪诊市场呈"大平台附带 + 个体散兵"的二元格局。春雨、京东健康、微医等综合平台将陪诊作为边缘功能，个体陪诊师则通过自媒体和微信接单。**医路安的机会在于做"陪诊垂类的美团"——标准化服务、透明定价、双向评价、平台保障。** 核心差异化：

1. **标准化**：三种服务类型明码标价，对比个体陪诊师 ¥200-800 的模糊定价，降低用户决策门槛
2. **信任体系**：认证展示 + 多维度评价 + 平台担保（责任险），解决个体接单的信任问题
3. **陪诊师赋能**：解决个体获客难题，提供稳定单源，这是供给侧的核心价值

### 用户分层策略

| 用户群 | 特征 | 产品策略 |
|--------|------|----------|
| 异地就医患者 | 对医院不熟、一次性需求 | 强调"当地人带你看病"、医院导航 |
| 老年人（及其子女） | 高频复诊、操作困难 | 代下单（家属端）、大字模式、复诊提醒 |
| 独居年轻人 | 偶发需求、价格敏感 | 半程陪诊/代办跑腿（低价入口）、社交分享 |
| 慢病患者 | 定期复诊、长期关系 | 服务套餐、指定陪诊师、随访提醒 |
| B 端客户（保险/企业） | 批量需求、API 对接 | B 端合作 API、企业健康管理方案 |

### 商业化路径

短期（MVP）：服务佣金模式（平台从每单抽成 15-20%），已有三种标准定价。中期：引入服务套餐/多次卡提升客单价和复购，会员体系提升 LTV。长期：B 端合作（保险增值服务、企业健康管理）打开第二增长曲线，药品代购代送拓展服务品类。

关键指标建议：
- **北极星指标**：月完成订单数（GMV）
- **供给侧**：活跃陪诊师数、平均接单响应时间、陪诊师留存率
- **需求侧**：下单转化率（浏览→下单）、复购率、NPS
- **体验**：平均评分、投诉率、取消率
