# 设计 Review - 2026-04-20

> 审核范围：wechat/ 小程序全部 28 个页面 + 9 个组件 + 全局样式体系
> 审核人：Design Lead (Claude)

---

## 1. 总体评价

医路安小程序在设计基础设施上做了不少正确的事：`styles/variables.wxss` 提供了约 60 个 CSS 自定义属性覆盖颜色、字号、间距、圆角、阴影、动画全维度；`app.wxss` 构建了按钮系统（primary / secondary / ghost / danger 四级）、卡片系统（三级阴影）、表单组件、工具类和骨架屏等全局 UI 原子。这是一个**有意识的设计系统胚胎**，但距离「被一致执行」还有明显差距。

核心矛盾：**token 定义了但没被贯彻**。pages/ 目录 27 个 wxss 文件中硬编码 hex 值共 423 处，components/ 目录 56 处。与此同时 pages/ 中 `var(--` 引用仅 726 处。也就是说约 **37% 的颜色声明绕过了 token**，一旦品牌色微调（例如 #1890FF → #2196F3）将需要手动逐文件替换。

整体风格定位——蓝绿渐变主色 + 白卡片 + 浅灰底——在医疗场景下合理：既专业又不冰冷。但**组件层面的执行一致性、无障碍覆盖、以及异形屏适配**是当前最大的体验风险。

---

## 2. 亮点

1. **Token 体系已成型** — `variables.wxss` 110 行涵盖品牌色、功能色、渐变、文字色、背景、边框、字号（6 级）、行高（3 级）、间距（7 级 8px 网格）、圆角（6 级）、阴影（5 级 + 品牌阴影）、动画、z-index，分类清晰。
2. **全局 UI 原子完备** — `app.wxss` 476 行提供 btn-primary/secondary/ghost/danger + btn-sm/btn-group、card/card-elevated/card-flat、form-group/form-label/form-input/form-textarea、badge 四种语义色、flex 工具类、gap 工具类、divider、skeleton 动画、empty-state、fade-in/slide-up 动画、safe-area 工具类。
3. **状态色梯度设计** — 订单状态通过渐变色区分（橙/蓝/绿/灰/红），视觉辨识度高。
4. **Safe Area 意识** — 固定底部栏均使用 `env(safe-area-inset-bottom)`，刘海屏适配到位。
5. **骨架屏** — 全局提供 skeleton 动画类，加载态体验优于简单 spinner。
6. **class 命名语义化** — kebab-case 统一，按功能而非视觉命名，可维护性好。

## 3. 关键问题

### P0 — 组件硬编码颜色严重（影响可维护性）

**问题**：9 个自定义组件中 **0 个** 使用 `@import` 引入 token 文件，且组件 wxss 无法直接继承 page 级 CSS 变量（微信组件隔离）。components/ 中 56 处硬编码 hex。`companion-card`（11 处）、`order-card`（12 处）、`service-card`（8 处）是重灾区。

**影响**：品牌色调整时组件颜色不随全局变化，造成视觉不一致。

**方案**：组件 JSON 中设置 `"styleIsolation": "apply-shared"` 或 `"shared"`，使组件可继承页面 CSS 变量，然后将硬编码值替换为 `var(--*)` 引用。

**排期**：1-2 天。

### P0 — 无障碍(a11y)几乎空白

**问题**：全部 28 个页面 + 9 个组件中，仅 `role-select` 页面出现 2 处 `role=` 属性。无 `aria-label`、无 `aria-live`、无 `aria-current`。

**影响**：屏幕阅读器用户完全无法使用。医疗 App 用户中不乏视障或老年人群，这是合规和道德风险。

**方案**：
- 所有可点击 view 加 `aria-role="button"`
- Tab bar 当前项加 `aria-current="page"`
- rating-stars 加 `aria-label="评分 {{value}} 星"`
- loading-overlay 加 `aria-busy="true"`
- 图标/emoji 加 `aria-label` 或 `aria-hidden`
- 动态内容区域加 `aria-live="polite"`

**排期**：2-3 天。

### P1 — 页面级 token 采纳率不均

**问题**：14 个页面（patient/home、patient/order-detail、companion-detail、login 等近期页面）已较好使用 `var(--*)`，但 13 个页面（companion/setup 59 处硬编码、profile/edit 43 处、delete-account 34 处、companion/order-detail 34 处）几乎纯硬编码。

**影响**：同一 App 内视觉风格存在「两层皮」，老页面和新页面微妙不一致。

**方案**：分批迁移，优先高流量页面（setup、profile/edit、order-detail），每次 PR 迁移 2-3 个页面。

**排期**：3-5 天。

### P1 — 重复 Tab Bar 组件

**问题**：`patient-tab-bar` 和 `companion-tab-bar` 的 WXML、WXSS、JS 代码 100% 相同，仅路由映射不同。

**影响**：维护成本翻倍，改一个忘改另一个。

**方案**：合并为单一 `role-tab-bar` 组件，通过 `properties.routes` 传入路由配置。

**排期**：0.5 天。

### P2 — Modal 遮罩不一致

**问题**：遮罩层透明度在不同页面间存在差异：`rgba(0,0,0,0.3)` / `0.45` / `0.5` / `0.6` 混用。Token 中定义了 `--color-bg-mask: rgba(0,0,0,0.45)` 但未被统一引用。

**方案**：所有 modal-mask 统一使用 `var(--color-bg-mask)`。

**排期**：0.5 天。

## 4. 设计 Token 缺口

Token 体系虽已成型，但仍缺少以下维度：

| 缺失 Token | 当前状态 | 建议补充 |
|------------|---------|---------|
| **组件尺寸** | 按钮高度在页面中硬编码 72/80/88/96rpx | `--size-btn-sm/md/lg` |
| **头像尺寸** | 36-180rpx 各自硬编码 | `--size-avatar-xs/sm/md/lg/xl` |
| **图标尺寸** | 无统一规范 | `--size-icon-sm/md/lg` |
| **边框宽度** | 1rpx/2rpx 混用 | `--border-width-thin/normal` |
| **字重** | 400/500/600/700 散落 | `--font-weight-normal/medium/semibold/bold` |
| **最大宽度** | 部分页面设 max-width，部分不设 | `--width-content-max` |
| **过渡属性** | 各页面自行组合 transition | `--transition-base` 快捷值 |

## 5. 组件一致性问题

| 组件类型 | 问题 | 严重度 |
|---------|------|--------|
| **Button** | 全局定义了 4 级按钮，但页面内大量自定义 `.btn-book`、`.btn-accept`、`.btn-submit` 等未复用全局类 | 中 |
| **Input** | 全局 `.form-input` 定义了 88rpx 高度、统一边框，但 companion/setup 和 profile/edit 中各自定义了不同高度和边框样式 | 中 |
| **Modal** | 无全局 modal 组件，每个页面自行实现 `.modal-mask` + `.modal-content`，结构和动画不统一 | 高 |
| **Toast** | 无全局 toast/notification 组件，各页面用 `wx.showToast` 或自定义实现 | 低 |
| **List Item** | `settings-item`、`menu-item`、`info-row` 等列表项在不同页面各自实现，高度/内边距/箭头图标不统一 | 中 |
| **Empty State** | 全局类和 `empty-state` 组件并存，组件内部硬编码颜色与全局类不一致 | 低 |

## 6. 体验细节缺失

1. **图标系统缺失** — 无统一 icon font 或 SVG sprite。当前使用 emoji（📋📊💬）做图标，在不同设备上渲染差异大，且无法控制颜色/尺寸。建议引入 icon font（如 iconfont.cn 自定义字体包）或统一 SVG 组件。

2. **触达区域过小** — rating-stars 每颗星区域约 44rpx（约 22px），低于苹果 HIG 建议的 44pt 最小触达区域。部分小按钮（badge、tag）同样偏小。

3. **色盲友好性不足** — 订单状态仅靠颜色区分（橙/蓝/绿/红），未搭配文字标签或图标。红绿色盲用户无法区分「进行中（绿）」和「已取消（红）」。

4. **暗色模式** — 当前无暗色模式支持。Token 体系为单主题。医疗 App 夜间使用频率不低（住院场景），建议 V2 规划。

5. **折叠屏/平板适配** — 所有页面 100% 宽度铺满，无 max-width 限制。在平板或折叠屏展开状态下，内容拉伸导致阅读体验差。

6. **加载态不统一** — 部分页面用全屏 loading-overlay，部分用骨架屏类，部分用 `wx.showLoading`，缺乏统一规范。

7. **动效层面** — 全局提供了 fadeIn/slideUp/skeleton 动画，但列表页缺少列表项交错入场动画（仅 available-orders 有 card-fade-in）。页面间转场依赖微信默认，无自定义过渡。

## 8. 设计系统规范化路线建议

### Phase 1: 修补（Sprint 1-2，约 1 周）
- 组件打开 styleIsolation，消灭组件内硬编码
- 合并重复 tab-bar
- 高频页面（setup、profile/edit、order-detail）token 迁移
- a11y 基础属性（aria-role、aria-label、aria-current）
- 补充缺失 token（组件尺寸、头像尺寸、边框宽度）

### Phase 2: 抽象（Sprint 3-4，约 1 周）
- 抽取全局 modal 组件（mask + content + 动画统一）
- 引入 icon font 替代 emoji
- 统一 list-item 组件
- rating-stars 触达区域扩大 + a11y 增强
- 订单状态增加图标辅助（色盲友好）

### Phase 3: 扩展（Sprint 5+）
- 暗色模式 token 集（`prefers-color-scheme` 或手动切换）
- 折叠屏 / 平板响应式布局（max-width + 多列）
- 组件文档（Storybook for 小程序 或自建 demo 页）
- 设计走查自动化（lint 规则检查硬编码 hex）

### 量化目标
- Token 采纳率从 ~63% → **95%+**（硬编码 hex < 30 处，仅限确需覆盖场景）
- a11y 属性覆盖：所有可交互元素具备 role + aria-label
- 组件复用率：全局组件覆盖 80%+ UI 模式，页面自定义样式 < 50 行
