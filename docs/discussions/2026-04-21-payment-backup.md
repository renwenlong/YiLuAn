# 2026-04-21 支付备份方案可行性 memo（A21-13）

**Owner：** Architecture / Backend
**关联：** B-01 微信支付商户号发布前置项
**Decision Due：** 2026-04-25
**性质：** 纯文档可行性 memo，不做技术原型

---

## 背景与目的

B-01 申请微信支付（直连）商户号是发布单一最大不确定项：执照已就绪但行业资质材料、银行卡 0.01 验证、特约商户审核都属于不可控外部时延，最坏情况会顶到首发窗口。架构师在 A21 晨会要求本周内产出**备份支付通道**的可行性 memo，避免一旦微信直连卡壳没有 Plan B。

本备份方案聚焦：**MVP 上线前后能否走第三方/聚合通道完成微信支付（小程序原生 `wx.requestPayment`）+ 退款**，并不替换最终的微信直连商户号长期路径。

候选三家：连连支付（连连数字）、Adyen、Pingpp（Ping++）。

---

## 候选 1：连连支付（连连数字）

- **公司定位：** 连连数字（LianLian DigiTech）持人民银行颁发的支付业务许可证（第三方支付牌照），主体业务覆盖**国内收单 + 跨境收付**两条线。海外品牌「LianLian Global」更偏跨境电商场景，国内业务由连连支付（lianlianpay.com）承载。MVP 关心的是国内收单。
- **小程序场景适配度：** 连连支付提供「微信小程序支付」服务商模式接入：商户在连连侧开通后，由连连作为微信支付服务商发起统一下单，前端仍走 `wx.requestPayment`，体验对终端用户透明。
- **对接难度：** 中。需要：(a) 在连连开户并提交医疗陪诊行业资质（与微信直连基本同套材料）；(b) 接入连连开放平台的下单/退款/回调三套接口；(c) 由于走服务商模式，前端 `wx.requestPayment` 的 `signType / paySign` 由连连返回。
- **费率：** 公开口径约 **0.6%（标准）**，部分行业可谈到更低；陪诊属于「生活服务」，预计 0.6%–1% 区间。需要走 BD 报价确认。
- **合规：** 央行牌照齐全，支持开发票，资金 T+1 结算入对公账户。
- **风险：** 服务商模式下回调链多一跳（连连 → 我方），故障域更大；如果未来切回微信直连，订单/支付历史的对账规则要兼容两套。

## 候选 2：Adyen

- **定位：** 全球收单平台，总部荷兰，主要服务跨国商户。
- **支持中国大陆？** **不能直接做中国大陆境内收单。** Adyen 文档明确支持 WeChat Pay 作为 *跨境* 支付方式（包括 WeChat Pay Mini Program 接入），但收单主体在境外，资金以外币结算到境外账户。其面向的典型客户是「海外品牌想收中国游客的微信付款」。
- **对接难度：** API 设计现代，文档质量高，对接成本相对低（人日意义上 5–8 人日完成沙箱→生产）。
- **费率：** Interchange++ 模型，跨境 WeChat Pay 一般在 **2.5%–3.5%** 区间（含跨境通道费），显著高于国内直连。
- **资质门槛：** 需要境外公司主体（一般是香港或新加坡），还要走 Adyen KYC / underwriting。我方目前为境内 WFOE/内资公司，**不满足主体条件**。
- **结论：** 对当前 MVP **不适用**。除非未来做出海版本，否则不进入候选短名单。

## 候选 3：Pingpp（Ping++）

- **当前状态：** Ping++ 于 2018 年被点融网（Dianrong）收购，公司主体仍在运营，官网（pingxx.com）和文档持续可访问，聚合支付 SaaS 产品仍在售。但行业普遍观感是**产品迭代节奏减缓**、社区活跃度下降，2024–2025 年没有重大产品发布。继续使用属于「能用但生态偏冷」。
- **定位：** 聚合支付抽象层。商户对接 Ping++ 一套 SDK/API，由 Ping++ 路由到微信支付/支付宝/银联等真实通道。
- **价值点：** 抽象层。如果未来要同时支持微信支付 + 支付宝 + 其他通道，省掉 N 套对接。
- **对接难度：** 低（其历史定位就是「降低支付对接成本」），3–5 人日可上线。
- **费率：** Ping++ 自身**不收取交易手续费**（按笔/月费 SaaS 计费，万级别），真实费率取决于底层通道——也就是说**仍然需要微信支付商户号**或其服务商商户号。Ping++ 不解决 B-01 的根本问题。
- **风险：** 多一层依赖；公司若进一步收缩，迁移成本要算。
- **关键判断：** 因为 Ping++ 不自带商户号，**不能作为 B-01 微信支付不到位时的真正备份**，只能作为「未来多通道抽象」的可选项。

## 结论与建议

### MVP 阶段建议路径

**主路径不变：继续推进 B-01 微信支付直连商户号**（费率最优、链路最短、零额外抽成）。

**备份路径（启用条件：B-01 在发布前 7 天仍未审核通过）：**
- **采用「连连支付服务商模式」**作为 Plan B。
  - 理由：是三家中**唯一同时满足「持牌 + 国内收单 + 小程序 wx.requestPayment + 主体匹配」**的方案。
  - 行动项：本周由 BD/法务先与连连开户经理建立联系、索取行业资质 checklist 和正式费率函；技术侧不动手对接，仅在 4/28 复盘 B-01 进度后决定是否启动 8–12 人日的对接工作。

**Adyen 排除**：境内主体不匹配；Mini Program 跨境收单费率过高，且资金落地在境外账户，不适合面向国内 C 端的陪诊业务。

**Pingpp 排除（短期）**：自身不解决商户号问题，无法作为 B-01 的备份。

### 长期建议

- 一旦 B-01 微信直连商户号到位，**所有真实流量走微信直连**；连连若已对接，保留为容灾通道（极小流量保活）。
- 12 个月内若产品要扩展支付宝/银联或多通道路由，再评估引入 **Pingpp 或自建抽象层**（自建已在 Phase 1 PaymentService 中预留接口，倾向自建以避免供应商锁定）。
- Adyen 仅在出海版本立项后再次评估。
