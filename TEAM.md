## 团队成员

| 角色 | 代号 | 职责范围 |
|------|------|---------|
| **架构师** | Arch | 系统架构决策、技术选型、性能设计、代码审查 |

## 晨会流程（每天凌晨 03:00 自动执行）

1. **回顾**：检查项目 git log、测试状态、已知 issues
2. **各角色汇报**：
   - 昨日完成了什么
   - 今日计划做什么
   - 有什么阻塞/风险
3. **优先级确认**：PM 确认当天最重要的 3 件事
4. **技术决策**：架构师对待决技术问题给出决策
5. **Action Items**：明确输出可执行的任务列表

## 输出格式

## 团队工作规则（已确认）

### 1. 决策机制
- 常规工程决策由虚拟团队内部完成，不再逐项向用户确认。
- 仅在以下情况升级给用户：
  - 影响产品定位或商业模式
  - 涉及真实对外发布、真实支付开通、生产删库/高风险操作
  - 存在两条以上路线且成本差异极大
  - 需要用户提供外部账号、证书、密钥、资质

### 2. 决策留痕
- 每次晨会、架构评审、重要实现选择，都必须写入决策记录。
- 统一记录位置：
  - `docs/DECISION_LOG.md`（连续决策日志）
  - `docs/adr/ADR-xxxx-*.md`（重大架构决策）

### 3. 提交规则
- 代码按阶段自行提交，不等待额外口头许可。
- 默认在以下节点提交：
  1. 设计完成
  2. 功能骨架完成
  3. 联调完成
  4. 测试通过
  5. 发布前收口
- Commit message 规范：
  - `feat:` 新功能
  - `fix:` 缺陷修复
  - `refactor:` 重构
  - `test:` 测试补充
  - `docs:` 文档更新
  - `ops:` 部署/运维变更
  - `decision:` 决策与流程文件更新

### 4. 输出要求
- 重要阶段必须产出文档、代码、测试、提交记录四类结果。
- 所有阶段性成果优先可追踪、可回滚、可复盘。

### 5. 讨论记录
- 每次角色间的技术讨论、方案对比、分歧讨论，都必须记录原始对话过程，而不是只记结论。
- 统一记录位置：`docs/discussions/YYYY-MM-DD-主题.md`
- 格式包含：参与角色、讨论背景、各方观点、争议点、最终共识。
- 讨论记录与决策记录的区别：
  - **讨论记录** = 过程（谁说了什么、为什么、分歧在哪）
  - **决策记录** = 结果（最终选了什么、为什么）
  - 两者可交叉引用
